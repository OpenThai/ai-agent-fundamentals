# บทที่ 6: Planning และ Reflection

---

Agent ที่ดีไม่ได้แค่ลงมือทำอย่างเดียว — ต้อง **วางแผน** ก่อนทำ และ **ตรวจสอบ** หลังจากทำ

สองความสำคัญนี้ทำให้ Agent แตกต่างจากระบบอัตโนมัติทั่วไป

---

## Planning: การวางแผน

### ทำไม Agent ต้องวางแผน?

LLM มีข้อจำกัด — ตอบได้เฉพาะสิ่งที่อยู่ใน context ถ้าให้ทำงานซับซ้อนโดยไม่วางแผน มันจะ:

- ตอบไม่ครบ
- ข้ามขั้นตอนสำคัญ
- ใช้เครื่องมือผิด
- ไม่รู้ว่าควรหยุดเมื่อไร

### รูปแบบ Planning

#### 1. Plan-then-Execute
วางแผนทั้งหมดก่อน แล้วลงมือตามแผน
```
แผน:
1. ค้นหาข้อมูลบริษัท (search_web)
2. อ่านรายงานการเงิน (read_pdf)
3. วิเคราะห์แนวโน้ม (analyze_data)
4. สรุปผล (generate_report)

→ ลงมือตามลำดับ
→ ถ้าขั้นไหนล้มเหลว หยุดและรายงาน
```

**เหมาะกับ**: งานที่ขั้นตอนชัดเจน ทำนายขั้นตอนได้

#### 2. Dynamic / Re-planning
วางแผนคร่าวๆ แล้วปรับระหว่างทาง
```
แผนเริ่มต้น: หาข้อมูลคู่แข่ง
└─ ระหว่างทางเจอว่าข้อมูลคู่แข่ง A หายาก → ปรับแผน
    └─ เปลี่ยนไป focus ที่คู่แข่ง B และ C แทน
```

**เหมาะกับ**: งานที่ข้อมูลไม่แน่นอน ต้องปรับแผนบ่อย

#### 3. Hierarchical Planning
แตกงานเป็นระดับ — งานใหญ่ → งานย่อย → งานย่อยลงไปอีก

```
วิจัยตลาด AI ในไทย
├─ ข้อมูลปริมาณตลาด
│  ├─ ค้นหาตัวเลขจากวิจัยกรุงศรี
│  └─ ค้นหาตัวเลขจาก IDC
├─ คู่แข่งในตลาด
│  ├─ รายชื่อ startup AI ไทย
│  └─ ผลิตภัณฑ์ของแต่ละราย
└─ แนวโน้มและโอกาส
   ├─ นโยบายรัฐบาล
   └─ เทคโนโลยีใหม่
```

**เหมาะกับ**: งานซับซ้อน หลายระดับ

## Reflection: การตรวจสอบตัวเอง

### Reflection คืออะไร?

คือกระบวนการที่ Agent **หันกลับมาตรวจสอบ** ผลลัพธ์ของตัวเอง

1. ทำงานเสร็จ
2. ตรวจสอบว่าถูกต้องไหม
3. ถ้าผิด → แก้ไข
4. ตรวจสอบอีกครั้ง
5. ทำซ้ำจนกว่าจะผ่าน

### รูปแบบ Reflection

#### 1. Self-check
Agent ถามตัวเอง: "คำตอบนี้ถูกต้องหรือยัง?"
```
Agent: ขายาว = 5, ฐาน = 3 → พื้นที่ = 5*3/2 = 7.5
  [ตรวจสอบ] 5*3 = 15 → 15/2 = 7.5 ✓
```

#### 2. Tool-assisted Check
ใช้เครื่องมือยืนยันผลลัพธ์
```
Agent คำนวณตัวเลขเสร็จ
  → เรียก calculator เพื่อยืนยัน
  → calculator ได้ค่าเดียวกัน → ✓
```

#### 3. Multi-pass
คิดหลายรอบ แล้วเทียบคำตอบ
```
รอบ 1: ตอบ A
รอบ 2: ตอบ A (เหมือนเดิม) → มั่นใจขึ้น
รอบ 3: ตอบ A → สรุป A

หรือ

รอบ 1: ตอบ A
รอบ 2: ตอบ B (ต่าง) → ต้องตรวจสอบเพิ่ม
```

#### 4. External Validator
ใช้ Agent อีกตัวหรือเครื่องมือตรวจสอบ
```
Agent A: เขียนโค้ด
Agent B: review โค้ดของ Agent A
Agent B: "เจอบั๊กในฟังก์ชัน calculate_tax()"
Agent A: แก้ไข
```

## เมื่อ Planning + Reflection ทำงานร่วมกัน

```
คุณ: "สร้าง report ยอดขาย Q1"

Agent:
  [PLAN] 1. query ฐานข้อมูล → 2. วิเคราะห์ → 3. สร้างตาราง → 4. export PDF
  [ACT] query ฐานข้อมูล: ได้ข้อมูลยอดขาย 3 เดือน
  [REFLECT] ข้อมูลครบไหม? → เช็ค: ม.ค. 600K, ก.พ. 750K, มี.ค. 820K ✓
  [ACT] วิเคราะห์: trends, YoY comparison
  [REFLECT] ตัวเลขสมเหตุสมผล? → เทียบกับ Q1 ปีก่อน → growth 12% → plausible ✓
  [ACT] สร้างตารางและ export PDF
  [REFLECT] PDF เปิดได้? file size ปกติ? → ✓
  [DONE] "report พร้อมแล้วครับ"
```

## ข้อจำกัดและความเสี่ยง

| ปัญหา | คำอธิบาย |
|-------|----------|
| **Over-planning** | วางแผนนานเกินไป ไม่ได้ลงมือสักที |
| **Planning hallucination** | วางแผนขั้นตอนที่ทำไม่ได้จริง |
| **False confidence** | ตรวจสอบแล้วคิดว่าถูก ทั้งที่ผิด |
| **Circular reflection** | แก้แล้วแก้ lại ไม่จนสักที |
| **Cost** | ยิ่ง reflection หลายรอบ ยิ่งแพง |
| **Time** | แต่ละรอบใช้เวลา LLM คิด |

### วิธีป้องกัน

- **Limit iterations** — จำกัดจำนวนรอบการ reflection
- **Timeout** — จำกัดเวลาสูงสุด
- **Clear success criteria** — รู้ว่า "พอ" เมื่อไร
- **Fallback to human** — ถ้า reflection ไม่จบ ให้ส่งให้มนุษย์

---

## ความเข้าใจผิดที่พบบ่อย

> "Planning ทำให้ Agent ทำงานช้าลง"

ช้าตอนแรก แต่ประหยัดเวลาโดยรวม เพราะลดการเดินผิดทาง

> "Reflection กัน Agent ผิดได้ 100%"

ไม่ใช่ครับ Reflection ลดความผิดพลาดได้ แต่ไม่ถึง 100% ต้องมี verification เพิ่มเติม

> "ยิ่ง reflection หลายรอบยิ่งดี"

ถึงจุดหนึ่ง diminishing returns — รอบที่ 3-4 คุณภาพไม่ต่างจากรอบแรกมาก แต่ cost เพิ่ม

---

## สรุป

- Planning ช่วยให้ Agent ทำงานเป็นระบบ ไม่หลงทาง
- Reflection ช่วยให้ Agent ตรวจสอบและแก้ไขตัวเอง
- Planning + Reflection = ทำงานอย่างมีคุณภาพ
- ต้องมีขีดจำกัด — ไม่ planning นานเกินไป ไม่ reflection หลายรอบเกินไป
- ยังไงก็ต้องมีมนุษย์ตรวจสอบอีกชั้น
